# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repository is

This is the **deployment repo** for Mini Platform, a GitOps Kubernetes reference
stack. It is not an application codebase — there is no service source code to
compile. It holds the environment-specific, first-party half of the stack that
**Argo CD** reconciles:

- `minikube/values/` — **integration overlays**, one `*-values.yaml` per managed
  release. This is where Mini Platform's customizations live. Contains **no
  credentials**.
- `minikube/gitops/` — the **Argo CD app-of-apps** wiring plus Vault secret
  mappings and ingress routes (the only first-party Helm charts here).
- `scripts/` — bootstrap, deploy, validate, and port-forward tooling.

`minikube/` is one **environment overlay**. A new environment is a sibling
directory (e.g. `prod/`) reusing the same charts repo.

The **vendored upstream Helm charts** (`charts/`) live in a **separate repo**,
[`mini-platform`](https://github.com/nolimitkun/mini-platform). Treat those as
third-party; they are not linted or modified here. Several scripts expect a
local checkout of that repo as a sibling directory (`../mini-platform`), override
with `--charts-dir` or the `CHARTS_DIR` env var.

## Two-source, multi-source model

Every generated Argo CD `Application` is **multi-source**: source 0 pulls the
chart (from the charts repo for `charts/*` paths, or from this repo for the
first-party `minikube/gitops/*` charts) and source 1 is a `ref: deploy` pointing
at this repo so each `valueFiles` entry can be written `$deploy/<path>`
(repo-root-relative, never depending on chart-path depth). This is the reason
the split works at all — a single-source Application cannot mix a chart from one
repo with a values file from another.

Two `(repoURL, revision)` pairs therefore drive everything and **must stay in
sync** across files: `chartsRepo`/`chartsRevision` and `deployRepo`/`deployRevision`.
Never hand-edit them in more than one place — run `scripts/set-repo.sh`.

## Two invariants that govern every change

1. **Argo CD reconciles Git commits, never the working tree.** Editing an
   overlay or GitOps chart has no effect until it is committed *and pushed* to
   the branch/revision Argo CD tracks. A clean `git status` (in **both** this
   repo and the charts checkout) is a precondition for the `--local-source`
   deploy path.
2. **No credentials in Git.** Vault is the source of truth for all application
   secrets; Vault Secrets Operator (VSO) materializes Kubernetes Secrets from
   it. Never add a Secret manifest or a literal credential to `minikube/values/`
   or `minikube/gitops/`. New credentials are generated by
   `scripts/bootstrap-vault-secrets.sh` and surfaced via
   `minikube/gitops/vault-resources/`.

## Validation (run before every commit to GitOps wiring)

```bash
CHARTS_DIR=../mini-platform ./scripts/validate-gitops.sh
```

This is exactly what CI (`.github/workflows/validate.yaml`) runs. It:

- checks every `valuesFile` referenced by the app-of-apps exists locally,
- checks every `chartPath` resolves (`charts/*` against `CHARTS_DIR`, skipped if
  that checkout is absent; first-party `minikube/gitops/*` locally),
- `helm lint`s and `helm template`s the three `minikube/gitops/` charts (passing
  all four source parameters),
- runs `kubeconform` over the rendered manifests, and
- `shellcheck`s `scripts/*.sh`.

`helm` is required; `kubeconform` and `shellcheck` are skipped if absent locally
but always run in CI. There is no test suite beyond this validator — "tests"
here means rendering and schema-checking.

## Architecture

Default AI request path: `Open WebUI → LiteLLM → vLLM Router → vLLM`, with
LiteLLM emitting traces to Langfuse. LiteLLM is an OpenAI-compatible gateway.

Supporting subsystems: Vault + VSO (secrets), Ingress NGINX (`.test`
hostnames), Prometheus + Grafana (metrics), MLflow (experiments), Qdrant
(vectors), Spark Operator, Superset → Trino (analytics), Keycloak (identity),
MinIO (S3 object store), all reconciled by Argo CD.

**Vault must be initialized and unsealed before dependent apps go healthy.**
Early Argo CD syncs intentionally show missing-secret failures until Vault is
seeded and the `VaultStaticSecret` resources synchronize — this is expected, not
a bug.

### The app-of-apps and sync waves

`minikube/gitops/root-application.yaml` is the single root Argo CD `Application`.
Its `source` points at this repo's app-of-apps chart
(`minikube/gitops/mini-platform/`) and passes four helm parameters
(`chartsRepo`, `chartsRevision`, `deployRepo`, `deployRevision`). That chart's
`templates/applications.yaml` loops over the `applications:` map in
`minikube/gitops/mini-platform/values.yaml` to emit one multi-source Argo CD
`Application` per release.

Each entry maps a `chartPath` (under `charts/` in the charts repo, or
`minikube/gitops/` here) to a `valuesFile` (under `minikube/`) and a **sync
wave** that orders rollout:

| Wave | Tier |
| --- | --- |
| `-5` | Argo CD |
| `-4` | Vault, VSO |
| `-3` | Vault secret mappings, stateful deps (postgres, redis, qdrant, minio, spark-operator) |
| `-2` | Keycloak, Langfuse, MLflow, Trino, vLLM |
| `-1` → `0` | Prometheus, Grafana, JupyterHub, Superset |
| `1` → `3` | LiteLLM, then Open WebUI, then ingress routes |

All generated Applications use `automated` sync with `prune: true`,
`selfHeal: true`, `CreateNamespace=true`, and `ServerSideApply=true`.

### Secret flow

`scripts/bootstrap-vault-secrets.sh` writes random credentials into the Vault
KV mount `mini-platform/`. `minikube/gitops/vault-resources/values.yaml` lists
secret *names*; its template renders a `VaultStaticSecret` per name that VSO
syncs into a like-named Kubernetes Secret. The two lists must agree: a secret a
workload needs must be **both** written by the bootstrap script **and** listed
in `vault-resources`. Note the shared `litellm-langfuse` secret — Langfuse's
headless init provisions the starter org/project from those keys and LiteLLM
reads the same keys for tracing, so no browser setup is needed.

## Common conventions / recipes

**Adding or changing a managed component:** edit the matching
`minikube/values/*.yaml` overlay (not the vendored chart — that lives in the
charts repo). To add a *new* release: add the upstream chart under `charts/` in
the **charts repo**, add an overlay under `minikube/values/`, add an
`applications:` entry in `minikube/gitops/mini-platform/values.yaml` with an
appropriate `wave` (`chartPath: charts/<name>`,
`valuesFile: minikube/values/<name>-values.yaml`), and (if it needs credentials)
add a secret to `bootstrap-vault-secrets.sh` and its name to
`minikube/gitops/vault-resources/values.yaml`. To expose it in a browser, add a
route in `minikube/gitops/ingress-resources/values.yaml`.

**Pointing at forks/branches:** the two source URLs/revisions appear in several
files that must stay in sync. Never hand-edit them — run:

```bash
./scripts/set-repo.sh \
  --charts-repo-url https://github.com/<owner>/mini-platform.git --charts-revision <branch> \
  --deploy-repo-url https://github.com/<owner>/mini-platform-deployment.git --deploy-revision <branch>
```

It rewrites `minikube/gitops/root-application.yaml`,
`minikube/gitops/mini-platform/values.yaml`, and the `scripts/deploy-minikube.sh`
defaults together. Any subset of the four values may be passed.

**Deploying / bootstrapping:** `scripts/deploy-minikube.sh` automates the full
Minikube + Argo CD + Vault bring-up (key flags: `--reset`, `--local-source`,
`--no-gpu`, `--rotate-secrets`, `--charts-dir`). It uses the local charts
checkout only to bootstrap Argo CD itself and, in `--local-source` mode, to
serve **both** repos through an in-cluster git daemon. The default
`minikube/values/vllm-values.yaml` requests an NVIDIA GPU — for CPU-only work,
edit that overlay or pass `--no-gpu`. `scripts/port-forward-services.sh` exposes
services on the Minikube host (`--mode lan` for LAN access).

The README is the authoritative deployment/operations reference and includes the
full manual bootstrap, service endpoints, and credential-retrieval commands.
Consult it before changing deploy/bootstrap behavior.

## Notes

- This is a **local reference stack**, not production: in-cluster TLS is off,
  Vault uses manual single-key unseal, and Trino/Prometheus/databases/vLLM are
  cluster-internal with no auth. Don't assume hardening that isn't there.
- `tools/` holds optional utility workloads (e.g. `net-tools.yaml`) applied
  manually, outside the GitOps reconciliation.
- `.github/workflows/validate.yaml` is the CI gate.
